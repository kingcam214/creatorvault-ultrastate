# CreatorVault Deployment Access Repair

## Agent Boundary: Do Not Ask the User for Access

This repair guide is for a trusted maintainer environment that already has deployment authority. Agents working on VaultX / CreatorVault must not ask the project owner for VPS credentials, SSH keys, GitHub secrets, passwords, or sync decisions. If the current execution environment lacks access, report it only as an internal deployment blocker and continue with code-side work. See `ops/DEPLOYMENT_ACCESS_LAW.md`.


This repository now includes the deployment script and the access repair script needed to stop the recurring GitHub Actions to VPS failure.

## What Is Fixed in the Repository

The production workflow now performs an explicit secret preflight before trying to SSH into the VPS. If the deploy key is missing, it fails with the exact missing secret name instead of the generic appleboy SSH error. The workflow also supports a temporary `CREATORVAULT_VPS_PASSWORD` fallback, but the durable path is still **passwordless SSH** through `CREATORVAULT_VPS_SSH_KEY`.

The missing `deploy_work_to_prod.sh` file is now committed. That removes the second fragile dependency where the workflow expected an untracked VPS-only script.

| Problem | Durable Fix |
|---|---|
| GitHub Actions had no usable VPS private key or password. | Use `ops/repair-github-vps-deploy-access.sh` to create a dedicated deploy key, install the public key on the VPS, and set the GitHub secrets. |
| Workflow did not say which secret was missing before starting SSH. | `.github/workflows/deploy.yml` now has a preflight step for `CREATORVAULT_VPS_HOST`, `CREATORVAULT_VPS_USER`, and either `CREATORVAULT_VPS_SSH_KEY` or `CREATORVAULT_VPS_PASSWORD`. |
| Workflow depended on `./deploy_work_to_prod.sh`, but the file was not in the repo. | `deploy_work_to_prod.sh` is now committed and owns PM2 start/reload behavior. |

## One Command to Repair Access

Run this from a trusted machine that already has SSH access to the VPS and a GitHub CLI login with permission to set repository secrets.

```bash
cd /path/to/creatorvault-ultrastate
chmod +x ./ops/repair-github-vps-deploy-access.sh
VPS_HOST="YOUR_VPS_IP_OR_HOSTNAME" VPS_USER="root" ./ops/repair-github-vps-deploy-access.sh
```

The script creates or reuses `~/.ssh/creatorvault_actions_deploy`, installs the public key in the VPS user’s `authorized_keys`, sets these GitHub Actions secrets, and verifies passwordless SSH:

| Secret | Purpose |
|---|---|
| `CREATORVAULT_VPS_HOST` | VPS hostname or IP address. |
| `CREATORVAULT_VPS_USER` | SSH user used by the deploy workflow. |
| `CREATORVAULT_VPS_SSH_KEY` | Private half of the dedicated deploy key generated by the repair script. |

## After Running the Repair

After the script reports `repair complete`, rerun the failed GitHub Actions deployment workflow for the current commit. The deployment will then SSH into `/root/creatorvault`, pull `main`, build, run `deploy_work_to_prod.sh`, and keep the PM2 app online.

## If the VPS Directory Is Different

If the production checkout is not `/root/creatorvault`, run the script with `REMOTE_PROJECT_DIR` for verification and update `.github/workflows/deploy.yml` accordingly before redeploying.

```bash
VPS_HOST="YOUR_VPS_IP_OR_HOSTNAME" VPS_USER="root" REMOTE_PROJECT_DIR="/actual/path" ./ops/repair-github-vps-deploy-access.sh
```
