The audit confirms that the platform already contains some correct ledger-backed work, especially in `activationWarRoomRouter`, where completed `transactions.status = 'completed'` rows drive first-dollar and recovery truth flags. The dangerous paths are not the abandoned-checkout recovery queue; the dangerous paths are the generic Empire Challenge and agent telemetry paths that can still turn test Stripe events, manual entries, or agent-estimated revenue into challenge progress.

| Risk area | File or surface | Current behavior | Required correction |
|---|---|---|---|
| Stripe webhook challenge credit | `server/_core/stripeWebhook.ts` | Credits the active challenge on `checkout.session.completed`, `payment_intent.succeeded`, `invoice.paid`, and standalone `charge.succeeded` based on amount alone. The code does not gate challenge credit by `event.livemode === true`, nor does it pass a durable proof packet into the challenge hook. | Only live-mode Stripe events with provider object id, customer reference, product/source, channel, and amount may credit real challenge revenue. Test-mode events must be recorded or ignored as non-revenue, never counted in `current_revenue`. |
| Universal challenge hook | `server/challengePaymentHook.ts` | Accepts amount, source, and description only, then directly inserts `empire_challenge_transactions` and increments `empire_challenges.current_revenue`. | Replace with a provenance-gated hook. The default must be no credit unless `mode === 'live'`, `proofId` exists, amount is positive, and source/channel/customer/product context is present. |
| Agent swarm automation | `server/routers/challengeAutomationRouter.ts` | Adds agent-estimated `revenue` from automation cycles and single-agent/full-cycle runs into `empire_challenges.current_revenue`. | Agent output revenue must remain telemetry/projection/opportunity only. It must not credit the challenge ledger. |
| Manual challenge revenue | `server/routers/challengeAutomationRouter.ts` `logChallengeRevenue` | Lets an authenticated user enter amount/source/description and increments challenge revenue directly. | Manual entry must be blocked from real challenge credit unless it carries completed payment proof. In the immediate patch, this route should return a proof-required response instead of mutating revenue. |
| AI Empire direct logging | `server/routers/aiEmpireRouter.ts` | `logRevenue` inserts challenge transactions and increments `current_revenue` from input amount. | Remove direct real-revenue mutation; treat this as proof-required unless wired to the same live-money hook. |
| Analytics display risk | `client/src/pages/king/ChallengeStoryEngine.tsx`, `agentTelemetryRouter`, AI empire pages | Displays `current_revenue` and agent revenue as if it represents collected money. | UI must label challenge progress as live-payment-backed only and separate projected/agent impact from collected cash. |
| Existing safer lane | `server/routers/activationWarRoomRouter.ts` and `client/src/pages/ActivationWarRoomCommandCenter.tsx` | Explicitly separates checkout starts/value from completed ledger revenue and uses truth flags such as `revenueIsLedgerBacked`. | Keep this pattern and reuse its language across challenge and revenue surfaces. |

The immediate implementation standard is therefore: **real revenue equals live-mode provider-confirmed payment evidence only**. Test payments, agent estimates, manual notes, checkout starts, projections, and abandoned checkout value may remain visible, but they must be named as test, estimated, attempted, queued, or recoverable value. They must not increase real challenge progress or collected-cash dashboards.
